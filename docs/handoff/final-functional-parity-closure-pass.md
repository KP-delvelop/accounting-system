# Final Functional Parity Closure Pass

Date: 2026-06-26

Scope: close the current local-first Bansi parity roadmap as far as practical in one Builder pass. This pass focuses on function, utility, workflow, accounting behavior, reports, exports, and testable golden outputs. It does not chase pixel-perfect Bansi UI.

Safety boundary: real Bansi remains read-only only. This pass did not perform new real-site mutation and did not store credentials, OTP, tokens, secrets, or real PII. Real identifiers remain omitted/redacted. Local demo data may be reset/generated by tests.

## Real Bansi Reference Used

No new real-site click-through was needed in this pass. I used the sanitized reference material already captured in:

- `docs/handoff/bansi-real-readonly-comparison-report.md`
- `docs/handoff/phase-14-bansi-functional-parity-golden-output.md`
- `docs/handoff/phase-15-bansi-functional-gap-map-v2.md`
- `docs/handoff/phase-16a-document-lifecycle-numbering-parity.md`

Relevant Bansi behaviors for this pass:

- Reports and accounting pages expose PDF/Excel/export-style controls.
- Trial balance, ledger-like accounting views, settlement/tax outputs, and transaction detail surfaces need traceability back to source documents/journals.
- Detail/voucher/ledger parity remains important, but a full voucher/detail-page redesign is larger than a safe final closure change.

## Triage Result

| Area | Decision | Reason |
| --- | --- | --- |
| Ledger/settlement/VAT export breadth | Fixed now | Low-risk UI/export improvement using the same report row models already rendered in the Reports screen. |
| Detail/voucher full pages | Backlog | Needs read-only detail templates, route/state selection, and print/voucher UX. Too broad for final compressed pass. |
| PDF/Excel native output | Backlog | Requires file-generation format decisions and visual QA; CSV remains safer and spreadsheet-friendly. |
| Import/migration workflow | Backlog | Needs field template policy and validation UX. Existing golden/test fixtures cover safe local fake data generation. |
| Settings/accounting policy | Backlog | Requires owner/accountant sign-off before changing tax, numbering, currency, and posting defaults. |
| Banking transfers/assets/inventory/payroll | Backlog | Separate modules, not a safe final-pass rewrite. |

## Remaining Differences / Backlog

| Gap | Blocker? | Recommendation |
| --- | --- | --- |
| Cancel/void/reversal accounting policy | Non-blocker for demo; blocker for production accounting parity | Define reversal and numbering reserve/skip rules with accounting owner before implementation. |
| Full transaction detail/voucher pages | Non-blocker | Add read-only detail route showing summary, movement, journal lines, attachments, audit, and ledger links. |
| Native PDF/Excel output | Non-blocker | Add one output type at a time, starting with Trial Balance or voucher once detail pages exist. |
| Import/migration UI/templates | Non-blocker | Create sanitized templates with explicit required fields and validation. |
| Bank transfers/reconciliation/fixed assets | Non-blocker for core prototype; blocker for full Bansi replacement | Split into banking transfer first, then reconciliation, then assets. |
| Inventory/warehouse/payroll | Non-blocker for current accounting prototype | Separate roadmap modules. |
| Production auth/DB/storage | Blocker for production | Choose provider/session architecture, hosted database, backup, storage, secrets, and retention policies. |
| Accounting UAT/sign-off | Blocker for production | Accountant/product-owner sign-off needed for VAT/WHT/FX/numbering rules. |
